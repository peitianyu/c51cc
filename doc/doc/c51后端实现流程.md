# C51 后端实现流程

本文给出基于 `ssa.h` 的 C51 后端落地方案，目标不是一次性覆盖全部 8051 特性，而是按照“**先跑通闭环，再逐步补齐能力**”的顺序推进实现。

文档重点回答 4 个问题：

1. 后端到底以什么数据结构和 ABI 为准。
2. 每个阶段要解决什么问题。
3. 每个阶段结束时，代码应达到什么可验证状态。
4. 哪些风险会反复出现，应该提前怎样约束。

---

## 1. 设计目标

- **统一输入**：后端仅消费 `ssa.h` 中定义的 `SSAUnit`、`Func`、`Block`、`Instr`、`GlobalVar` 等结构。
- **分阶段交付**：每个阶段都能独立验收，避免“大而全”导致长期不可运行。
- **契约优先**：先明确支持子集，不支持的特性必须显式报错，不能静默退化。
- **面向 C51 约束**：实现过程始终围绕 8051/C51 的寄存器、地址空间、调用约定和指令特性展开。
- **便于演进**：先支持最小整数闭环，再逐步引入全局对象、phi 消除、ABI、地址空间、寄存器分配和平台特化能力。

---

## 2. 全局约束

### 2.1 目标机假设

| 项目 | 约定 |
|------|------|
| 架构 | C51 / 8051 |
| 字节序 | 大端序 |
| 默认内存模型 | `large` |
| 默认指针宽度 | 2 字节 |
| 指针表示 | 高字节在前、低字节在后 |
| 多字节整型 | `int` 为 2 字节补码 |

### 2.2 表示约定

- `bool`：单字节 `0/1`。
- `char`：单字节。
- `int`：2 字节，大端序存放。
- 指针：2 字节地址值，大端序存放。
- 寄存器对：若用 `R6:R7` 表示 16 位值，则 `R6` 为高字节，`R7` 为低字节。

### 2.3 实现原则

1. **测试先行**：阶段测试在开发前准备好，作为验收基线。
2. **约束贯穿**：大端序、地址空间、ABI 必须在所有模块中保持一致。
3. **不支持即报错**：所有超出当前阶段范围的输入，都要在生成入口尽早拒绝。
4. **先正确再优化**：先确保指令选择和控制流语义正确，再做寄存器分配和 peephole 优化。

---

## 3. 后端总流程

推荐按下面的总链路组织后端：

1. **契约检查**：确认输入 SSA 落在当前支持子集内。
2. **函数/全局预处理**：收集符号、类型、地址空间和初始化信息。
3. **指令选择**：将 SSA 指令映射为 C51 抽象目标指令。
4. **CFG 落地**：确定基本块布局，消除 `phi`，补齐边拷贝。
5. **ABI 降级**：落实参数传递、返回值和调用点保存。
6. **寄存器分配**：将虚拟值映射到实体寄存器或 spill 槽。
7. **目标特化**：按地址空间和平台特性选择 `MOV/MOVX/MOVC/JB/...` 等指令。
8. **汇编优化与输出**：做局部 peephole，输出汇编/HEX 或后续链接输入。

这 8 步并不要求一开始全部完成，但后续每个阶段都应尽量贴近这个总流程，避免返工。

---

## 4. 阶段路线图

下面的阶段划分遵循“**最小闭环优先**”原则。每个阶段都包含：目标、关键点、交付物、验收测试。

---

## 阶段 0：定义后端契约

**目标**：明确第一批支持的 SSA 子集，并建立统一的拒绝策略。

### 关键点

- 支持类型：`bool`、`char`、`int`、指针。
- 类型信息来自 `Ctype`，至少要能获取：
  - 基本类型
  - 宽度
  - 有无符号
  - 地址空间
- 暂不支持：
  - 结构体/联合体传参
  - 复杂返回值
  - 浮点
  - 中断函数
  - 复杂初始化
- 首批支持的 SSA 指令建议包括：
  - 算术/逻辑：`IROP_ADD/SUB/AND/OR/XOR/NOT/NEG`
  - 移位：`IROP_SHL/SHR`
  - 比较：`IROP_EQ/NE/LT/LE/GT/GE`
  - 转换：`IROP_TRUNC/ZEXT/SEXT`
  - 指针：`IROP_ADDR/OFFSET/INTTOPTR/PTRTOINT`
  - 内存：`IROP_LOAD/STORE`
  - 控制流：`IROP_JMP/BR/RET/CALL/PHI`
  - 选择：`IROP_SELECT`
- 在 `c51_gen.c` 入口统一检查：
  - `SSAUnit` 是否包含不支持的全局对象或函数属性
  - `GlobalVar` 初始化形式是否受支持
  - `Func.is_interrupt` 是否为假

### 交付物

- `c51_gen.c` 中的契约检查逻辑。
- 一份支持子集说明文档，例如 `docs/c51_supported_features.md`。

### 验收测试

`test/stage0/contract.c`

```c
int add(int a, int b) { return a + b; }
int main() { return add(1, 2); }
```

预期：

- 该程序编译成功。
- 若输入结构体参数、中断函数等不支持特性，应直接报错。

## 阶段 1：整数最小闭环

**目标**：在“单函数、无全局”的范围内，完成整数运算、分支和返回。

### 关键点

- 遍历 `Func.blocks`，按块顺序生成汇编。
- 支持基础算术、逻辑、比较、移位和常量加载。
- 比较结果统一物化成 `0/1`，不要跨块依赖 PSW 标志。
- 多字节整数运算必须显式处理高低字节及进位/借位。
- `A` 寄存器只作短生命周期临时值使用，分支前要落回普通位置。

### 交付物

- 函数发射主流程。
- 算术/逻辑/比较/移位的指令选择模块，例如 `c51_isel_arith.c`。
- 控制流指令选择模块，例如 `c51_isel_ctrl.c`。

### 验收测试

`test/stage1/arith.c`

```c
int arith(int a, int b) {
    int x = a + b;
    int y = a - b;
    return (x ^ y) + (x & y);
}
int main() { return arith(9, 4); }
```

`test/stage1/branch.c`

```c
int branch(int a, int b) {
    if (a > b) return 11;
    if (a == b) return 22;
    return 33;
}
int main() { return branch(5,3)+branch(4,4)+branch(1,7); }
```

## 阶段 2：全局对象与基础内存访问

**目标**：支持全局变量、取地址、解引用和简单偏移访问。

### 关键点

- 从 `SSAUnit.globals` 中提取 `GlobalVar` 信息。
- 根据 `is_static`、`is_extern`、`has_init` 等属性生成符号和段信息。
- `IROP_ADDR` 负责生成符号地址。
- `IROP_OFFSET` 负责常量偏移的地址计算。
- `IROP_LOAD/STORE` 至少支持 1 字节和 2 字节对象。
- 阶段 2 可以先只覆盖 `data` 空间，其他空间留到阶段 5。

### 交付物

- 全局对象生成模块，例如 `c51_gen_global_var.c`。
- 基础内存访问指令选择模块，例如 `c51_isel_mem.c`。

### 验收测试

`test/stage2/global.c`

```c
int g = 7;
int inc() { g = g + 1; return g; }
int main() { return inc() + inc(); }
```

`test/stage2/ptr.c`

```c
int x = 3;
int *p = &x;
int main() { *p = *p + 5; return x; }
```

`test/stage2/offset.c`

```c
int sum() {
    char arr[3];
    arr[0]=1; arr[1]=2; arr[2]=3;
    return arr[0]+arr[1]+arr[2];
}
int main() { return sum(); }
```

## 阶段 3：CFG 落地与 `phi` 消除

**目标**：让循环和分支汇合路径正确落地，不再依赖 SSA 形式执行。

### 关键点

- 基本块建议按逆后序或接近 fallthrough 的顺序发射。
- `phi` 消除采用前驱边拷贝策略。
- 多个 `phi` 同时出现时，需要并行 move 以打破循环依赖。
- 16 位值复制时必须保持高字节/低字节顺序一致。
- `phi` 降级完成后，应从 `Block.phis` 中移除，不再参与后续生成。

### 交付物

- `c51_isel_ctrl.c` 中的边拷贝逻辑。
- 并行 move 实现。

### 验收测试

`test/stage3/if_phi.c`

```c
int merge(int a, int b, int c) {
    int x;
    if (a > b) x = a + c;
    else x = b + c;
    return x + 1;
}
int main() { return merge(7,3,2) + merge(1,9,2); }
```

`test/stage3/while_phi.c`

```c
int sum_n(int n) {
    int i=0, s=0;
    while (i < n) { s += i; i++; }
    return s;
}
int main() { return sum_n(6); }
```

`test/stage3/parallel_move.c`

```c
int swap_like(int a, int b, int flag) {
    int x = a, y = b;
    if (flag) { x = b; y = a; }
    return x - y;
}
int main() { return swap_like(8,3,0) + swap_like(8,3,1); }
```

## 阶段 4：函数调用 ABI

**目标**：稳定支持参数传递、返回值和调用保存规则。

### 关键点

- 参数类型来自 `Func.params` / `param_types`。
- 可先采用以下最小 ABI：
  - `char`：`R7`
  - `int` / 指针：`R6:R7`
  - 超出寄存器容量的参数：按从右到左压栈
- 返回值：
  - `char`：`R7`
  - `int` / 指针：`R6:R7`
- 调用点要显式处理跨调用活跃值。
- 嵌套调用场景中，必要的临时值必须 spill 到内存。

### 交付物

- `IROP_CALL` 的 lowering。
- 调用点保存与被调保存逻辑。

### 验收测试

`test/stage4/call_basic.c`

```c
int add1(int x) { return x + 1; }
int main() { return add1(10); }
```

`test/stage4/call_args.c`

```c
int add3(int a, int b, int c) { return a + b + c; }
int main() { return add3(1,2,3); }
```

`test/stage4/call_nested.c`

```c
int inc(int x) { return x + 1; }
int add2(int a, int b) { return a + b; }
int main() { return add2(inc(3), inc(4)); }
```

`test/stage4/ptr_param.c`

```c
int g;
void set_ptr(int *p) { g = *p; }
int main() { int x=5; set_ptr(&x); return g; }
```

## 阶段 5：多地址空间完整化

**目标**：覆盖 `data`、`idata`、`xdata`、`code`、`sfr`、`sbit` 等空间，并按空间生成正确访存指令。

### 关键点

- 在类型系统中显式携带地址空间信息。
- 全局对象按空间进入对应 section。
- 指针本身仍是 16 位地址值，但“访问该地址时用什么指令”由目标空间决定。
- 访存指令映射建议：
  - `data` / `idata`：`MOV`
  - `xdata`：`MOVX`
  - `code`：`MOVC`
- `sbit` 走位操作指令，例如 `SETB`、`CLR`、`MOV C, bit`、`JB`、`JNB`。

### 交付物

- 按地址空间分配全局对象的逻辑。
- 区分空间的 `LOAD/STORE` 指令选择。
- `sbit` 的读写与分支支持。

### 验收测试

`test/stage5/spaces.c`

```c
unsigned char data  d = 1;
unsigned char idata i = 2;
unsigned char xdata x = 3;
unsigned char code  c = 4;
int main() { return d + i + x + c; }
```

`test/stage5/space_ptr.c`

```c
unsigned char data  d = 0x11;
unsigned char xdata x = 0x22;
unsigned char data *p1 = &d;
unsigned char xdata *p2 = &x;
int main() { return *p1 + *p2; }
```

`test/stage5/sfr_bit.c`

```c
__sfr __at(0x90) P1;
__sbit __at(0x90) P1_0;
int main() {
    P1 = 0;
    P1_0 = 1;
    if (P1_0) return P1;
    return 0;
}
```

## 阶段 6：寄存器分配与 spill 增强

**目标**：减少不必要的内存往返，稳定处理高寄存器压力场景。

### 关键点

- 可以先做块内贪心，再升级为线性扫描。
- 16 位值必须按寄存器对分配，不能拆成无关联的两个单字节寄存器。
- `A` 保持“短暂占用、立即释放”的策略。
- spill 槽优先放在 `data` 区，统一由后端管理。
- `phi` 边拷贝、调用点保存和寄存器分配三者要协同，不可各自独立实现。

### 交付物

- 改进的寄存器分配器，例如 `c51_isel_regalloc.c`。
- 自动插入 spill / reload 的逻辑。

### 验收测试

`test/stage6/reg_pressure.c`

```c
int reg_pressure(int a, int b, int c, int d) {
    int v1 = a + b;
    int v2 = b + c;
    int v3 = c + d;
    int v4 = v1 ^ v2;
    int v5 = v2 ^ v3;
    int v6 = v4 + v5;
    return v6 + v1 + v3;
}
int main() { return reg_pressure(1,2,3,4); }
```

`test/stage6/lifetime.c`

```c
int lifetime(int p, int q, int r) {
    int keep = p + q;
    int t1 = r - p;
    int t2 = q ^ r;
    t1++; t2++;
    return keep + t1 + t2;
}
int main() { return lifetime(7,5,3); }
```

`test/stage6/cross_block.c`

```c
int cross(int a, int b, int c) {
    int x = a + b;
    int y = b + c;
    if (x > y) x += c;
    else y += a;
    return x + y;
}
int main() { return cross(3,4,5); }
```

## 阶段 7：C51 特化能力

**目标**：加入中断、内联汇编和位操作优化等平台专属能力。

### 关键点

- 中断函数依赖 `Func.is_interrupt`、`interrupt_id`、`bank_id` 等属性。
- 中断序言/尾声应保存 `A`、`PSW`、`DPTR` 以及被破坏的寄存器组，尾声使用 `RETI`。
- `IROP_ASM` 作为黑盒指令保留，不在内部做重排或优化。
- 平台模式匹配可逐步加入：
  - `if (bit)` → `JB/JNB`
  - 递减循环 → `DJNZ`
  - 比较跳转 → `CJNE`

### 交付物

- 中断专用序言/尾声生成。
- 内联汇编保留逻辑。
- 若干 C51 特化模式匹配。

### 验收测试

`test/stage7/sbit_branch.c`

```c
__sbit __at(0x90) flag;
int main() {
    flag = 1;
    if (flag) return 5;
    return 9;
}
```

`test/stage7/interrupt.c`

```c
volatile int tick;
void timer0_isr(void) __interrupt(1) {
    tick++;
}
int main() { return tick; }
```

`test/stage7/inline_asm.c`

```c
int add1(int x) {
    __asm__("INC R7");
    return x;
}
int main() { return add1(3); }
```

## 阶段 8：目标优化与编码收口

**目标**：在语义稳定的前提下，输出更干净的汇编和最终 HEX。

### 关键点

- peephole 优化仅针对局部、可证明安全的模式。
- 不得跨越以下边界擅自优化：
  - `volatile` 访问
  - 内联汇编
  - 特殊寄存器/位寄存器操作
- 输出层要统一 section 顺序、符号信息和重定位描述。

### 交付物

- 局部优化器，例如 `c51_optimize.c`。
- 编码和输出模块，例如 `c51_encode.c`、`c51_output.c`。

### 验收测试

`test/stage8/peephole.c`

```c
int redundant(int a) {
    int x = a;
    int y = x;
    int z = y;
    return z;
}
int main() { return redundant(6); }
```

预期：冗余搬运被删除。

`test/stage8/jump_fold.c`

```c
int nested_if(int a) {
    if (a > 0)
        if (a > 2) return 1;
    return 0;
}
int main() { return nested_if(3); }
```

预期：多余跳转被折叠。

`test/stage8/e2e.c`

```c
unsigned char data d = 1;
unsigned char xdata x = 2;
int helper(int p) { return p + d; }
int main() { return helper(x) + d + x; }
```

预期：端到端结果正确，并能稳定输出目标文件/HEX。

## 5. 推荐推进顺序

如果目标是尽快得到“能跑简单程序”的版本，建议按下面顺序推进：

`阶段 0 → 阶段 1 → 阶段 2 → 阶段 3 → 阶段 4 → 阶段 8（最小输出）`

这样可以先拿到：

- 基础整数运算
- 基本控制流
- 全局变量与指针
- 简单函数调用
- 可读汇编输出

之后再并行增强：

- `阶段 5`：多地址空间
- `阶段 6`：寄存器分配优化
- `阶段 7`：中断、位操作、内联汇编

## 6. 模块职责建议

为避免职责交叉，建议后端按下面方式组织：

| 模块 | 主要职责 |
|------|----------|
| `c51_gen.c` | 契约检查、函数级生成流程、串联后端各阶段 |
| `c51_gen_global_var.c` | 全局对象、符号、段与初始化信息生成 |
| `c51_isel_arith.c` | 算术、逻辑、比较、移位等指令选择 |
| `c51_isel_mem.c` | `LOAD/STORE`、地址计算、按空间访存 |
| `c51_isel_ctrl.c` | 分支、跳转、返回、phi 边拷贝、调用降级 |
| `c51_isel_regalloc.c` | 寄存器分配、spill、reload |
| `c51_optimize.c` | peephole 与局部目标优化 |
| `c51_encode.c` / `c51_output.c` | 汇编编码、目标输出、HEX 生成 |

这不是强制目录结构，但职责边界应尽量保持稳定。

## 7. 核心风险与应对

| 风险 | 典型表现 | 应对策略 |
|------|----------|----------|
| 字节序混淆 | 16 位值高低字节颠倒 | 所有 16 位测试加入 `0x1234`、`0x00ff`、`0xff00` 等样例 |
| 指针访问空间错误 | `xdata` 被当成 `data` 访问 | 在阶段 5 中分别覆盖 `MOV`、`MOVX`、`MOVC` 的测试 |
| phi 消除错误 | 循环变量或分支汇合值异常 | 强化回边、交换、并行 move 测试 |
| ABI 被破坏 | 嵌套调用返回值或参数错乱 | 阶段 4 与阶段 6 共用多参数、嵌套、指针传参样例 |
| spill 插入不正确 | 高寄存器压力下结果漂移 | 为跨块 live range 和调用点 live range 增加专项测试 |
| 优化误改语义 | 开优化后结果变化 | 保留 `-O0` 路径，对比优化前后程序行为 |
| 输入结构不匹配 | 生成阶段访问非法字段或遗漏属性 | 统一以 `ssa.h` 为准，禁止后端私自假设隐藏字段 |

## 8. 落地建议

如果按工程可执行性排序，建议团队在实现时持续检查下面 3 件事：

1. **当前阶段能否单独跑通测试**。
2. **新增类型/指令/地址空间后，契约检查是否同步更新**。
3. **大端序、ABI、地址空间这三项约束是否在新代码里保持一致**。

一句话总结：

> 先把“正确的最小 C51 后端”跑起来，再逐步把它做成“完整、稳定、可优化的 C51 后端”。
