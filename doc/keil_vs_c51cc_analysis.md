## 二、C51CC 正确且优于 Keil 的情形

### 2.1 常量折叠 / 死代码消除更彻底

C51CC 的 SSA+常量传播在很多简单用例中能直接折叠掉所有运算，输出最短代码，而 Keil 仍然生成完整的运算序列。

**典型案例**（C51CC 正确折叠为 `return 0`）：
- `00011`：`int x = y = 0; return x;` → C51CC 直接 `MOV R7,#0`，Keil 还写内存
- `00012`：`return (2+2)*2-8;` → 两者都折叠为 0，C51CC 代码更短（无 `CLR A` 中间步骤）
- `00022`：枚举值为 0 → C51CC 正确折叠
- `00054`：enum 类型 → C51CC 正确折叠

**代码体积提升**：常量折叠后 C51CC hex 普遍比 Keil 小 50~800 字节（包含了 Keil 的启动代码开销）。

### 2.2 直接内存寻址（无需运行时库辅助函数）

Keil 使用 `?C?ILDPTR` / `?C?ISTPTR` 等运行时库函数做指针间接读写，C51CC 通过静态分析直接展开为 `MOV Ri, addr` / `MOV addr, Ri`，避免了库调用开销。

**典型案例**：
- `00013`：`int *p = &x; return p[0];` → C51CC: `MOV R1, x` / `MOV R0, (x+1)`，Keil: `LCALL ?C?ILDPTR`
- `00014`：`*p = 0` → C51CC 直接 `MOV x, #0`，Keil: `LCALL ?C?ISTPTR`
- `00016`：数组元素指针 → C51CC 直接偏移访问
- `00039`：`void*` 转型后解引用 → C51CC 展开为直接内存比较

### 3.2 编译成功但仍需继续核对语义正确性的项

这一节的旧清单已部分过时：`00035`、`00036` 这类纯常量路径在当前源码中会被**正确**折叠为 `return 0`，不再属于错误优化；`00041` 也不再是“错误常量折叠”，而是寄存器分配/比较物化造成的代码质量问题。

当前更值得继续跟踪的语义正确性问题主要集中在：

1. **运行时 `struct` / `union` 访问边界**
    - 代表用例：`00018`, `00042`
2. **`void*` / 多级指针相关类型规则**
    - 代表用例：`00112`, `00144`
3. **复杂初始化与函数地址/lvalue 场景**
    - 代表用例：`00095`, `00149`, `00150`, `00151`

### 3.3 代码质量问题（寄存器分配 / spill）

#### (A) 过度 spill 到 XDATA（00041）

**案例 `00041`（素数筛）**：历史快照中，C51CC 生成了 6 个 `__spill_N` XDATA 变量，大量使用 `MOVX @DPTR` / `MOVX A, @DPTR`，hex 体积为 `1341` 字节；本轮通过保留比较分支，已下降到 `1263` 字节，但相比 Keil 的 `854` 字节仍大 **409 字节**（约 `+48%`）。  
Keil 使用 `DPTR`、`R2/R3`、`R4/R5` 等寄存器对高效管理，完全在片上 RAM 内完成。

**原因**：当前线性扫描寄存器分配算法在循环嵌套深度较大时，过早将活跃变量 spill 到 XDATA，而 MCS-51 的 XDATA 访问代价远高于寄存器。

#### (B) 不必要的寄存器移动（00034, 00007, 00008）

- `00034`（break/continue 循环）：C51CC 使用了 4 对寄存器（R0-R5），每次循环迭代都有额外的 `MOV Rx, Ry` 传递，Keil 只用 1 对（R6/R7）
- `00008`（do-while）：C51CC 每次迭代先复制再自减（`MOV R3,R1; DEC R3`），Keil 直接原地 `DEC R7`
- `00007`（for+while 混合）：C51CC 跳转标签结构与 Keil 基本等价，但有冗余 `MOV` 对

#### (C) code 段字符串加载冗余（00025, 00026）

- `00025`：C51CC 用 `MOV DPTR,#.L0; MOV R1,DPL; MOV R0,DPH` 转存指针再调用 `_strlen`，且调用前后有冗余的寄存器保存/恢复
- `00026`：字符串索引偏移 `+0` 的 `ADD A,#0` 未被消除，且 `MOV R6,A` 后立即 `MOV R6,#0`（死写）

#### (D) 比较指令生成冗余（00039 等）

`!=` 的通用“布尔值物化”仍有压缩空间；不过本轮已经避免把 `if (cmp) return 1; return 0;` 过早改写成 `return cmp`，使 `00041` 的 `ne v34, const 669` 恢复为 `CJNE ... ; LJMP` 的直接比较分支，更接近 Keil 的代码形态。

## 五、需要优化与实现的优先级清单

### 优先级 P0（影响正确性，大量用例失败）

> 当前状态：本轮重新核对的 P0 代表用例已全部转绿；剩余失败已主要落在 P1/P2/P4 类问题。

1. **乘法/除法/取模运算展开**
   - MCS-51 硬件只有 8×8 无符号乘法（`MUL AB`）
   - 需实现 16×16 有符号/无符号乘法展开函数（内联或运行时库）
   - 需实现 16÷16 有符号/无符号除法/取模（对应 Keil `?C?IMUL`/`?C?SIDIV`）
   - 影响用例：`00009`, `00036`(乘), `00041` 等

2. **复合赋值运算符 `+=` `-=` `*=` `/=` `%=` `<<=` `>>=` `&=` `|=` `^=`**
   - 当前实现导致赋值副作用被优化掉（常量折叠过激）
   - 影响用例：`00036`, `00101`, `00116`, `00117`, `00118` 等

3. **局部数组 / 局部 struct / union 的栈内存分配**
   - 需在函数帧中正确分配 DATA 段局部内存（类似 Keil 的 `?DT?func?file` SEGMENT）
   - 目前局部 struct/array 被忽略导致编译结果全错
   - 影响用例：`00015`, `00017`, `00018`, `00042`, `00043` 等

4. **`switch/case/default` 语句**
    - 已补齐深层嵌套 `case/default`、共享 body + label 降低，以及 Duff's device 风格控制流
    - 已修复用例：`00051`, `00143`

5. **逻辑非 `!` 和双重逻辑非 `!!` 运算符**
   - 影响用例：`00035`, `00116`~`00118`

### 优先级 P1（解析器特性）

6. **局部 struct/enum 类型定义（作用域隔离）**
   - 函数体内的 `struct T { ... }` 定义应与外部作用域隔离
   - 影响用例：`00044`, `00052`, `00053`

7. **多级指针 `**pp`**
   - 类型系统需支持任意深度指针类型解析和代码生成
   - 影响用例：`00019`, `00020`, `00038`, `00144`

8. **自引用结构体指针（递归类型）**
   - `struct S { struct S *p; }` 需要前向引用支持
   - 影响用例：`00019`

9. **函数指针类型**
   - 包括 `int (*fp)()` 变量声明、struct 中的函数指针字段、函数指针作为返回值
   - 影响用例：`00087`, `00088`, `00124`

10. **`short` / `long` / `unsigned short` / `unsigned long` 类型**
    - 扩展类型系统，支持 8-bit `char`、16-bit `short`、32-bit `long`
    - 影响用例：`00086`, `00111` 等

### 优先级 P2（语义正确性）

11. **Tentative definition（暂定定义）**
    - `int x; int x = 3; int x;` 在 C 标准中合法，需支持同名全局变量多次声明
    - 影响用例：`00095`, `00096`, `00110`

12. **Struct 指针字段访问（`->`）的正确代码生成**
    - 目前 `p->field` 在指针为运行时值时结果错误
    - 影响用例：`00018`, `00027`~`00029`

13. **Union 类型语义（共享内存布局）**
    - 影响用例：`00042`

14. **全局数组初始化列表**
    - `S a[1] = {{1, {2, 3}}}` 嵌套初始化列表
    - 影响用例：`00091`

### 优先级 P3（代码质量优化）

15. **寄存器分配优化——减少 XDATA spill**
    - 问题：深度嵌套循环中过早 spill，大量 XDATA 访问
    - 目标：尽量将活跃区间短的变量保留在 R0-R7 中
    - 影响用例：`00041`（生成代码比 Keil 大 58%）

16. **冗余 MOV 消除（copy propagation 后处理）**
    - `MOV R3,R1; MOV R2,R0` 后立即 `DEC R3` 可直接对原寄存器操作
    - 影响用例：`00008`, `00034` 等循环体

17. **16 位自增/自减优化**
    - 当前：`INC Rn; CJNE Rn,#0,skip; INC Rm` （3条）
    - 优化目标：等价于 Keil 的 `INC R7; CJNE R7,#00H,skip; INC R6`，逻辑相同但注释和结构可优化
    - Keil 在某些情况下使用 `DEC` 原地操作不需要额外 copy

18. **`!=` 比较指令压缩**
    - 已完成一部分：比较后直接返回的路径会保留为分支，避免额外布尔值物化
    - 剩余工作：把更多非分支 `ne` 场景压缩到 `XRL/ORL/JZ/JNZ` 风格

19. **Code 段指针 `+0` 偏移消除**
    - `MOV A,R1; ADD A,#0; MOV R1,A` 是冗余的，`+0` 偏移应在编译时消除
    - 影响用例：`00026`

20. **死写消除（dead store elimination）**
    - `00026` 中 `MOV R6,A`（subexpr 结果）后立即被 `MOV R6,#0` 覆盖，前者是死写

### 优先级 P4（标准库支持）

21. **`<stdio.h>` 最小化支持（`printf` 等）**
    - 30+ 个用例因缺 `<stdio.h>` 而无法编译
    - 可提供 MCS-51 UART 输出的简化 `printf` 实现

22. **预处理器 `##` token-paste 运算符**
    - 影响用例：`00141`

23. **`#line` 预处理指令**
    - 影响用例：`00152`

## 六、Keil 与 C51CC 代码风格差异

| 方面 | Keil | C51CC |
|------|------|-------|
| 函数入口标签 | `main:` | `_main:` |
| 段声明 | `RSEG ?PR?main?file` | 无（平坦输出） |
| 运行时库 | 链接 `?C_STARTUP`, `?C?IMUL`, `?C?SIDIV` 等 | 无（内联展开） |
| 注释风格 | Keil 源行注释 | SSA IR 操作注释 |
| 16位返回值约定 | R6=高字节, R7=低字节 | R6=高字节, R7=低字节（一致） |
| 参数传递 | R6/R7（第一参数）, R4/R5（第二参数） | 同（基本一致） |
| 局部变量 | 尽量分配到寄存器对，否则 DATA 段 | 寄存器优先，溢出到 XDATA |
| 字面量字符串 | `?CO?file SEGMENT CODE` + `?SC_N:` | `.LN:` + 直接 `DB` |
| hex 文件 | 含启动代码（+50字节基础开销） | 仅代码段 |

## 七、参考案例——寄存器溢出对比（00041）

### Keil（简洁，使用 DPTR 作 `t` 变量）
```asm
MOV     DPTR,#02H       ; t = 2
?C0003:
  MOV R7,DPL            ; t*t 乘法准备
  INC DPTR              ; t++
  SJMP ?C0003
```

### C51CC（大量 XDATA spill）
```asm
MOV DPTR, #__spill_1    ; 写 XDATA
MOV A, #2
MOVX @DPTR, A
...
MOV DPTR, #__spill_1    ; 读 XDATA
MOVX A, @DPTR
```

**结论**：C51CC 寄存器分配需专门针对 MCS-51 的 DPTR 寄存器做优化，避免将可放在 DPTR 的指针值 spill 到 XDATA。
